# LifeTracker v1.4.1 QA Checklist

本文件用于执行 `docs/TASKS.md` 中 T75.57-T75.59 的外部验收。不要把 token、邮箱、密码或 Supabase service role 写入本文档或提交到仓库。

## 1. 前置条件

- 后端运行在 `http://localhost:3020`。
- 前端运行在 `http://localhost:3021`。
- Supabase 已执行最新版 `docs/database-init.sql`。
- 至少准备两个测试账号，并拿到各自 Bearer token。
- 测试账号 A 已有一个可共享的物品或待办。

## 2. 自动化优先：社交/共享全链路

脚本位置：`scripts/qa/v141-social-flow.mjs`。

它会模拟两个用户完成登录、搜索用户、发送好友申请、接受申请、好友置顶/取消置顶、创建测试物品/待办、共享物品、查看对话、发送消息、标记已读，并在结束时清理测试数据。

建议使用一组专门的干净 QA 账号。脚本会自动清理 accepted 好友关系和 pending 申请；如果两个账号之间已存在 rejected 记录，需要在数据库中删除该 `life_friendships` 记录或更换 QA 账号后再跑。

```bash
export API_BASE="http://localhost:3020"
export QA_USER_A_EMAIL="user-a@example.com"
export QA_USER_A_PASSWORD="password-a"
export QA_USER_B_EMAIL="user-b@example.com"
export QA_USER_B_PASSWORD="password-b"

node scripts/qa/v141-social-flow.mjs
```

默认会清理脚本创建的测试物品、待办、共享和好友关系。调试时如需保留数据：

```bash
QA_KEEP_DATA=1 node scripts/qa/v141-social-flow.mjs
```

脚本通过后，可作为 T75.57 的主要验收依据；下面的 curl 步骤保留为定位问题时的手工拆解。

## 3. 手工 curl 验证

建议在本地 shell 中用环境变量保存测试参数：

```bash
export API_BASE="http://localhost:3020"
export TOKEN_A="user-a-access-token"
export TOKEN_B="user-b-access-token"
export USER_B_ID="user-b-uuid"
export ITEM_ID="owned-item-id"
export TODO_ID="owned-todo-id"
```

### 3.1 用户搜索

```bash
curl -sS "$API_BASE/api/messages/users/search?q=test" \
  -H "Authorization: Bearer $TOKEN_A"
```

预期：返回候选用户列表，不包含当前用户 A。

### 3.2 发送好友申请

```bash
FRIENDSHIP_ID=$(
  curl -sS "$API_BASE/api/messages/friends/requests" \
    -H "Authorization: Bearer $TOKEN_A" \
    -H "Content-Type: application/json" \
    -d "{\"target_user_id\":\"$USER_B_ID\",\"message\":\"你好，我想添加你为好友\"}" \
  | node -e 'let s="";process.stdin.on("data",d=>s+=d);process.stdin.on("end",()=>{const r=JSON.parse(s);console.log((r.data||r).id)})'
)
echo "$FRIENDSHIP_ID"
```

预期：创建 `pending` 好友申请，并返回 `id`。

### 3.3 查看好友申请

```bash
curl -sS "$API_BASE/api/messages/friends/requests" \
  -H "Authorization: Bearer $TOKEN_B"
```

预期：用户 B 能看到来自用户 A 的 incoming 申请。

### 3.4 接受好友申请

```bash
curl -sS "$API_BASE/api/messages/friends/requests/$FRIENDSHIP_ID" \
  -X PATCH \
  -H "Authorization: Bearer $TOKEN_B" \
  -H "Content-Type: application/json" \
  -d '{"action":"accept"}'
```

预期：申请状态变为 `accepted`，并自动创建或复用好友对话。

### 3.5 好友列表、置顶、取消置顶

```bash
curl -sS "$API_BASE/api/messages/friends" \
  -H "Authorization: Bearer $TOKEN_A"

curl -sS "$API_BASE/api/messages/friends/$FRIENDSHIP_ID/pin" \
  -X PATCH \
  -H "Authorization: Bearer $TOKEN_A" \
  -H "Content-Type: application/json" \
  -d '{"pinned":true}'

curl -sS "$API_BASE/api/messages/friends/$FRIENDSHIP_ID/pin" \
  -X PATCH \
  -H "Authorization: Bearer $TOKEN_A" \
  -H "Content-Type: application/json" \
  -d '{"pinned":false}'
```

预期：好友列表包含用户 B；置顶状态只影响当前操作用户视角。

### 3.6 共享给已通过好友

```bash
curl -sS "$API_BASE/api/shares" \
  -H "Authorization: Bearer $TOKEN_A" \
  -H "Content-Type: application/json" \
  -d "{\"resource_type\":\"item\",\"resource_id\":\"$ITEM_ID\",\"shared_with_id\":\"$USER_B_ID\",\"permission\":\"view\"}"
```

预期：创建 share 记录，返回 `conversation_id`，消息对话中出现资源卡片。

如果验证待办共享，将 `resource_type` 改为 `todo`，`resource_id` 改为 `$TODO_ID`。

### 3.7 对话和消息流

```bash
CONVERSATION_ID=$(
  curl -sS "$API_BASE/api/messages/conversations" \
    -H "Authorization: Bearer $TOKEN_A" \
  | node -e 'let s="";process.stdin.on("data",d=>s+=d);process.stdin.on("end",()=>{const r=JSON.parse(s);const list=r.data||r;console.log(list[0]?.id||"")})'
)

curl -sS "$API_BASE/api/messages/conversations/$CONVERSATION_ID" \
  -H "Authorization: Bearer $TOKEN_A"

curl -sS "$API_BASE/api/messages/conversations/$CONVERSATION_ID/messages" \
  -X POST \
  -H "Authorization: Bearer $TOKEN_A" \
  -H "Content-Type: application/json" \
  -d '{"type":"text","content":"接口验收消息"}'

curl -sS "$API_BASE/api/messages/conversations/$CONVERSATION_ID/read" \
  -X PATCH \
  -H "Authorization: Bearer $TOKEN_B"
```

预期：对话列表、消息列表、发送消息、标记已读均成功。

### 3.8 删除好友

```bash
curl -sS "$API_BASE/api/messages/friends/$FRIENDSHIP_ID/delete" \
  -X PATCH \
  -H "Authorization: Bearer $TOKEN_A"
```

预期：好友关系删除；后续再次共享给该用户应返回权限错误。

## 4. T75.58 Web PWA 截图 QA

启动服务：

```bash
cd backend && npm run start:dev
cd frontend && npm start -- --port 3021
```

浏览器打开 `http://localhost:3021`，使用稳定测试账号登录。至少截图并检查：

- 首页：问候语、统计卡、快捷入口、最近待办、通知红点。
- 工作台：核心入口、管理入口、生活记录、数据与提醒分组。
- 物品：搜索、筛选、列表卡、左滑删除确认、新增/编辑表单。
- 待办：分段筛选、优先级、截止时间、完成态、新增/编辑表单。
- 消息：好友申请、好友列表、置顶/删除、聊天页、共享资源卡片。
- 我的：账号资料、偏好、数据与支持入口。
- 低频页：分类、位置、模板、借用、日历、统计、通知、数据、资产、小组件。

每张截图检查浅色/深色背景、SafeArea、按钮文字、空态、弹窗和底部操作栏是否有遮挡或溢出。

## 5. T75.59 Android Development Build 冒烟

前置：使用 EAS Development Build，不使用 Expo Go。

建议冒烟路径：

- 登录、退出登录、刷新后保持会话。
- Tab 切换：首页、工作台、消息、我的。
- 物品和待办：创建、编辑、搜索、筛选、左滑删除、应用内确认弹窗。
- 图片选择、相机、剪贴板、分享、通知权限在 Android 上有降级或正常反馈。
- 键盘弹出时，新增/编辑页底部操作栏不遮挡输入项。
- 深色模式下 SafeArea 和页面底部不露白。
- 消息好友申请和共享资源卡片可正常打开。

完成后回填 `docs/TASKS.md`：将对应 T75.57-T75.59 改为 `done`，并把父任务 T75 从 `blocked` 改为 `done`。
