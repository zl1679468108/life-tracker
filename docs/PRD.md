## 1. 产品定位

<a id="1-product-positioning"></a>

LifeTracker 是一款**个人生活管理工具**，帮助用户追踪物品位置、管理待办事项，通过分类和位置体系实现生活条理化。

**核心价值主张**:
- 找东西不再翻箱倒柜——记录物品存放位置
- 待办不再遗漏——设置优先级和提醒
- 数据可视化——通过统计了解自己的生活模式

**竞品定位**: 介于备忘录类应用（Notion、备忘录）和物品管理类应用之间，轻量级、开箱即用、无需配置。

## 2. 用户画像

<a id="2-user-persona"></a>

### 主要用户群体
- **个人用户**，需要管理日常物品和任务
- **生活爱好者**，喜欢记录和整理
- **忙碌上班族**，需要快速记录待办和物品位置

### 使用场景
| 场景 | 描述 |
|------|------|
| 找东西 | 忘记某件物品放在哪里，通过搜索快速定位 |
| 物品整理 | 搬家或整理房间后，更新物品位置 |
| 任务管理 | 创建待办事项，设置优先级和截止日期 |
| 数据统计 | 回顾物品和待办的使用情况 |

## 4. 功能详细需求

<a id="4-feature-details"></a>

### 4.1 用户认证

#### 4.1.1 邮箱密码登录
- **需求**: 用户使用邮箱和密码登录
- **输入**: 邮箱、密码
- **输出**: 登录成功进入首页，失败显示错误提示
- **异常**: 账号不存在/密码错误时给出明确提示
- **会话**: 登录态持久化，下次打开应用自动登录

#### 4.1.2 邮箱密码注册
- **需求**: 新用户通过邮箱和密码注册
- **输入**: 邮箱、密码（确认密码可选）
- **输出**: 注册成功自动登录，失败显示错误提示
- **异常**: 邮箱已注册时提示

#### 4.1.3 密码重置
- **需求**: 忘记密码时可申请重置
- **流程**: 输入邮箱 → 发送重置邮件 → 用户点击邮件链接重置密码

#### 4.1.4 OAuth 第三方登录
- **需求**: 支持第三方平台登录（如微信、Google）
- **流程**: 点击登录 → 跳转授权页 → 回调 → 自动登录
- **当前状态**: 基础框架已实现，具体 provider 待配置

#### 4.1.5 登出
- **需求**: 用户可安全退出登录
- **行为**: 清除本地 token，跳转到登录页
- **平台差异**: Web 端使用原生 confirm 弹窗，移动端使用原生 Alert

#### 4.1.6 路由守卫
- **未登录**: 自动跳转到登录页
- **已登录**: 访问认证页面自动跳转到首页

### 4.3 物品管理

#### 4.3.1 物品列表
- 显示所有物品的列表
- 每张卡片显示: 分类图标、名称、描述、位置标签、分类标签
- 左滑显示删除按钮
- 下拉刷新

#### 4.3.2 搜索
- 支持搜索物品名称和描述
- 300ms 防抖延迟
- 聚焦时搜索框高亮（主色边框）
- 有内容时显示清除按钮
- 回车触发搜索

#### 4.3.3 分类筛选
- Chips 横向滚动
- 预设分类: 全部、电子产品、书籍、日用品、衣物、其他
- 选中状态高亮
- 支持用户自定义分类的筛选

#### 4.3.4 排序
- 底部弹出选择器
- 排序选项:
  - 添加时间（默认，最新在前）
  - 名称（A-Z）
  - 分类
- 选中项高亮显示

#### 4.3.5 批量模式
- 点击多选图标进入批量模式
- 显示批量操作栏（全选、已选数量、删除按钮）
- 每个物品卡片显示复选框
- 选中卡片有主色边框高亮
- 点击全选/取消全选
- 批量删除需要二次确认

#### 4.3.6 创建/编辑物品
- **必填字段**:
  - 物品名称（左图标 tag）
  - 分类（网格选择，带图标）
  - 存放位置（网格选择，带图标）
- **选填字段**:
  - 图片（最多 5 张，支持拍照/相册）
  - 描述（多行文本）
- **表单验证**:
  - 名称为空时提示"请输入物品名称"
  - 未选择分类时提示"请选择分类"
  - 未选择位置时提示"请选择存放位置"
  - 实时清除错误提示
- **保存按钮**: 导航栏右侧"保存"按钮
- **创建和编辑复用**: 同一页面，通过 URL 参数区分

#### 4.3.7 物品详情
- 顶部: 分类渐变图标 + 名称 + 描述
- 信息卡片: 分类和位置（图标 + 名称）
- 图片: 横向滚动，点击可全屏预览
- 时间: 添加时间和更新时间
- 编辑按钮: 导航栏右侧
- 删除按钮: 底部红色按钮，需要确认

#### 4.3.8 图片处理
- 选择: 拍照或从相册选择
- 压缩: 原生平台压缩至 1024px 宽，质量 0.8
- Web 端跳过压缩（不支持 ImageManipulator）
- 上传: 调用后端 API 上传到 Supabase Storage
- 预览: 全屏查看，支持滑动切换
- 删除: 点击图片预览中的删除按钮移除

### 4.8 通知中心

#### 4.8.1 通知列表
- **待办提醒**: 显示未完成的待办（前3条），包含截止日期
- **物品统计**: 显示当前物品总数
- **系统通知**: 欢迎消息等
- **推送通知**: 待办提醒时间到达时自动添加到通知列表
- **已读/未读状态**:
  - 未读通知有橙色浅背景 + 橙色边框高亮
  - 未读通知标题加粗，右侧显示红色圆点
  - 点击未读通知自动标记为已读
- **全部标记已读**: 未读 tab 下显示"全部标记为已读"按钮

#### 4.8.2 Tab 筛选
- **全部**: 显示所有通知
- **未读**: 仅显示未读通知
- **已读**: 仅显示已读通知
- Tab 切换使用分段控制器样式

#### 4.8.3 首页铃铛动画
- 有未读通知时，铃铛图标执行抖动动画（左右摇摆 2 次）
- 推送通知到达时，铃铛重新触发抖动动画
- 有未读通知时，铃铛右上角显示红色圆点
- 无未读通知时，铃铛静止且无红点

#### 4.8.4 实时推送通知（Socket）
- **后端调度器**: `ReminderScheduler` 服务每分钟检查一次 `life_todos` 表中 `reminder_date` 已到且未完成的待办
- **防重复机制**: 使用内存 Set 记录已发送的提醒（`todo.id + reminder_date`），避免重复推送
- **Socket 事件**: 后端通过 `reminders:fired` 事件向对应用户房间推送提醒数据
- **前端监听**: `notificationStore` 监听 `reminders:fired` 事件，自动将提醒添加到通知中心列表并触发铃铛抖动
- **连接管理**: 用户登录后 `socketService.connect(userId)` 加入 `user:${userId}` 房间，登出时断开

#### 4.8.5 数据持久化
- 已读状态通过 AsyncStorage 本地持久化
- 应用重启后保持已读状态

#### 4.8.6 空状态
- 无通知时显示铃铛关闭图标 + "暂无通知"
- Tab 筛选无结果时显示对应提示（"暂无未读通知" / "暂无已读通知"）

### 4.11 消息模块（v1.1.0 新增，v1.2.0 社交化增强）

#### 4.11.1 对话列表
- **对话来源**:
  - 分享自动创建：用户 A 分享物品/待办给用户 B → 自动创建双人对话
  - 手动发起：用户在消息列表页点击 FAB，搜索用户邮箱或从已有共享关系中选择
  - 系统通知：到期提醒、状态变更等聚合到"系统通知"对话
- **对话列表项**: 头像（对方用户）+ 最后一条消息摘要 + 时间 + 未读数
- **消息类型图标**: 物品卡片消息显示 📷 图标，待办显示 📝，文字显示 💬，系统显示 🔔
- **下拉刷新**
- **FAB 按钮**: 右下角悬浮添加按钮，点击打开"新建对话"面板

#### 4.11.2 新建对话面板
- **搜索用户**: 输入邮箱或用户名，模糊匹配已有用户
- **好友列表**: 从已有共享关系的用户中选择（快速发起）
- **首条消息**: 可选发送文字消息或快速分享物品/待办（从列表选择）
- **空状态**: 无搜索结果时显示引导文案

#### 4.11.3 对话详情
- **物品卡片消息**: 缩略图 + 名称 + 位置 + 有效期 + "查看详情"按钮（点击跳转物品详情）
- **待办卡片消息**: 标题 + 优先级标签 + "标记完成"按钮
- **文字消息**: 气泡样式，区分发送者/接收者
- **系统通知消息**: 灰色背景，无发送者气泡
- **输入框**: 文字消息输入 + 发送按钮 + 快速分享物品/待办入口
- **消息排序**: 按 created_at 升序（最新消息在下）

#### 4.11.3 分享联动
- 分享流程改造：
  ```
  物品详情页 → 点击分享 → 输入对方邮箱 →
    1. 创建 share 记录
    2. 创建 conversation（双方 participant_ids）
    3. 创建 item/todo 类型消息（卡片数据）
    4. 将 conversation_id 写入 share 记录
  ```
- 对方收到通知后点击 → 打开 app → 直接进入消息 Tab → 进入该对话

#### 4.11.4 搜索用户 API
- **接口**: GET /api/users/search?q=xxx
- **参数**: q — 搜索关键词（邮箱前缀或用户名）
- **返回**: 用户列表 `{ id, email, display_name, avatar_url }`
- **用途**: 新建对话面板中搜索目标用户

#### 4.11.5 手动创建对话 API
- **接口**: POST /api/messages/conversations/manual
- **参数**: `{ participant_ids: string[], initial_message?: { type, content, card_data } }`
- **返回**: 新对话对象 + 可选的首条消息
- **用途**: 用户手动发起与好友的对话

#### 4.11.6 数据结构

```typescript
interface Message {
  id: string;
  conversation_id: string;
  sender_id: string;
  type: 'item' | 'todo' | 'text' | 'system';
  resource_type?: 'item' | 'todo';
  resource_id?: string;
  content?: string;            // 文字消息内容
  card_data?: Record<string, any>;  // 卡片数据（名称、图片URL等）
  created_at: string;
}

interface Conversation {
  id: string;
  participant_ids: string[];     // 双人对话
  last_message_type?: string;
  last_message_content?: string;
  last_message_at?: string;
  created_at: string;
  updated_at: string;
}
```

#### 4.11.5 数据库表

```sql
CREATE TABLE IF NOT EXISTS life_conversations (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  participant_ids UUID[] NOT NULL,
  last_message_type VARCHAR(20),
  last_message_content TEXT,
  last_message_at TIMESTAMPTZ,
  created_at TIMESTAMPTZ DEFAULT NOW(),
  updated_at TIMESTAMPTZ DEFAULT NOW(),
  CHECK (array_length(participant_ids, 1) = 2)
);

CREATE TABLE IF NOT EXISTS life_messages (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  conversation_id UUID REFERENCES life_conversations(id) ON DELETE CASCADE,
  sender_id UUID NOT NULL REFERENCES auth.users(id),
  type VARCHAR(20) NOT NULL,
  resource_type VARCHAR(20),
  resource_id UUID,
  content TEXT,
  card_data JSONB,
  created_at TIMESTAMPTZ DEFAULT NOW()
);

CREATE INDEX IF NOT EXISTS idx_messages_conversation ON life_messages(conversation_id, created_at DESC);
CREATE INDEX IF NOT EXISTS idx_messages_sender ON life_messages(sender_id);

-- shares 表新增 conversation_id 关联
ALTER TABLE life_shares ADD COLUMN IF NOT EXISTS conversation_id UUID REFERENCES life_conversations(id);
```

## 6. 交互规范

<a id="6-interaction-specs"></a>

### 6.1 页面导航
- Tab 导航: 底部 4 个 Tab（首页、工作台、消息、我的）
- 工作台: 功能卡片矩阵（3 列网格），点击卡片跳转二级功能页面
- 二级页面: 物品列表页(/item/list)、待办列表页(/todo/list)等，含完整搜索/筛选/排序
- Stack 导航: 详情页、创建页作为子页面推入导航栈
- 返回按钮: 左上角返回箭头
- 手势返回: 支持侧滑返回（原生平台）

### 6.2 操作反馈
- **成功**: Toast 提示，绿色，1.5 秒后自动消失
- **错误**: Alert 弹窗，需要用户确认
- **加载**: Loading 遮罩或骨架屏
- **确认**: Alert 弹窗（删除、退出等敏感操作）

### 6.3 手势交互
- 左滑删除: 物品/待办列表项
- 下拉刷新: 所有列表页
- 侧滑返回: 子页面
- 点击复选框: 快速切换待办完成状态

### 6.4 空状态
- 所有列表页必须有空状态引导
- 包含图标、标题、描述、操作按钮
- 按钮引导用户创建第一条数据

### 6.5 表单交互
- 必填字段标记红色星号
- 实时表单验证，聚焦时清除错误
- 提交时全局验证
- 保存按钮在导航栏右侧
- 键盘自动处理（KeyboardAvoidingView）

### 6.6 加载状态
- 首次加载: 骨架屏（3 条）
- 刷新: 下拉刷新指示器
- 提交: 按钮 loading 状态

## 8. 技术约束

<a id="8-technical-constraints"></a>

### 8.1 平台适配
- 所有功能必须同时适配 **Web (PWA)** 和 **Android**
- 不能使用仅单端可用的方案
- Web 端不支持: `expo-image-manipulator`（压缩）、`expo-notifications`（推送）
- Web 端需使用替代方案

### 8.2 数据库约束
- 所有表使用 `life_` 前缀
- 开发和生产连接同一张表
- 所有表必须启用 RLS
- 分类/位置的 `user_id` 为 NULL 表示系统预设

### 8.3 认证约束
- 认证操作使用 anon key
- service_role key 仅用于管理员操作
- Token 存储在本地（AsyncStorage / localStorage）

### 8.4 API 约束
- 所有接口（除登录/注册）需 Bearer Token 认证
- user_id 由后端从 token 自动注入
- 错误统一格式: `{ message: string }`

### 8.5 文件上传约束
- 单文件 / 批量上传（最多 10 个）
- 存储桶: `items-images` (public)
- 路径格式: `{user_id}/{timestamp}_{filename}`
- Web 端跳过图片压缩

### 8.6 性能约束
- 列表数据考虑分页和缓存
- 搜索使用 300ms 防抖
- 图片上传前压缩（原生平台）

### 8.7 环境变量约束
- 开发环境需清空 `ICUBE_PROXY_HOST`（避免 Supabase 连接超时）
- 敏感信息不得硬编码

### 8.8 表名规范
- 后端表名使用复数: `life_categories`, `life_locations`
- 前端类型名使用单数: `LifeCategory`, `LifeLocation`
