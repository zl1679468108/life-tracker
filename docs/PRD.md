## 1. 产品定位

<a id="1-product-positioning"></a>

LifeTracker 是一款**个人生活管理工具**，帮助用户追踪物品位置、管理待办事项，通过分类和位置体系实现生活条理化。

**核心价值主张**:
- 找东西不再翻箱倒柜——记录物品存放位置
- 待办不再遗漏——设置优先级和提醒
- 数据可视化——通过统计了解自己的生活模式

**竞品定位**: 介于备忘录类应用（Notion、备忘录）和物品管理类应用之间，轻量级、开箱即用、无需配置。

## 2. 用户画像

<a id="2-user-persona"></a>

### 主要用户群体
- **个人用户**，需要管理日常物品和任务
- **生活爱好者**，喜欢记录和整理
- **忙碌上班族**，需要快速记录待办和物品位置

### 使用场景
| 场景 | 描述 |
|------|------|
| 找东西 | 忘记某件物品放在哪里，通过搜索快速定位 |
| 物品整理 | 搬家或整理房间后，更新物品位置 |
| 任务管理 | 创建待办事项，设置优先级和截止日期 |
| 数据统计 | 回顾物品和待办的使用情况 |

## 3. 功能总览

<a id="3-feature-overview"></a>

### 3.1 当前问题梳理与标准化原则

#### 现存问题

| 问题 | 表现 | 处理原则 |
|---|---|---|
| 视觉美观度不足 | 首页和工作台仍接近默认浅色卡片风格，层级不突出 | 首页/工作台统一深色操作台风格，强调高频动作和关键数据 |
| 入口职责重复 | 首页统计、快捷操作、工作台、设置二级页均存在交叉入口 | 首页只保留总览和高频动作；工作台只作为功能入口；复杂列表留在独立列表页 |
| 模块边界不清 | 物品/待办既是 Tab 又是二级管理入口，用户路径不稳定 | 物品/待办统一从工作台、首页快捷动作进入列表/新增/编辑流程 |
| 功能堆叠 | 工作台承载过多说明、列表或管理内容会造成重复 | 工作台禁止承载业务列表，仅展示入口图标和名称 |
| 异常状态表达分散 | 空状态、加载、错误提示在各页面风格不完全一致 | 列表页使用统一加载/错误/空态组件，首页只展示有价值的摘要 |
| 低频功能分散 | 统计、通知、借用、模板、共享、日历、分类、位置在多处重复出现 | 统一收拢到单一入口组，不允许在首页和工作台重复铺开 |

#### 标准化原则

- **四 Tab 定位固定**: 首页=今日总览，工作台=功能入口，消息=好友/通知/协作沟通，我的=账号与系统设置。
- **高频动作前置**: 添加物品、添加待办放在首页快捷操作；通知保留为头部图标；其他功能直接平铺在工作台分组里。
- **去冗余入口**: 同一功能可以有高频快捷入口和工作台入口，但页面内不重复展示同一管理列表。
- **页面不嵌套页面**: 首页不承载完整物品/待办列表；工作台不承载表单、列表、统计图表或详情内容。
- **权限一致**: 登录后访问业务数据；未登录统一拦截到登录页；好友申请通过后才能建立聊天和配置对该用户的共享权限。
- **单一职责**: 首页只看，工作台只进，列表页只管，设置页只配，不在同一页面混放多类任务。

### 3.2 本轮改版范围

#### 3.2.1 保留范围
- 物品管理
- 待办管理
- 消息对话
- 账号与系统设置
- 主题、语言、同步
- 反馈建议
- 数据管理

#### 3.2.2 收拢范围
- 统计
- 通知
- 借用
- 模板
- 共享（不做独立管理入口，集中到消息好友操作和资源编辑上下文）
- 日历
- 分类
- 位置
- 资产总览
- 桌面小组件
- 备份恢复
- AI 物品识别

#### 3.2.3 入口规则
- 首页仅保留总览、搜索、消息提醒和两个高频动作。
- 工作台仅保留功能入口分组，不承载业务数据和说明性内容。
- 我的页外层直接平铺账号与安全、偏好设置、数据与支持入口；不再把语言、密码、退出等常用设置藏到更深节点。
- 低频能力统一在工作台按分组平铺，不在首页和我的页重复展示。
- 每个功能必须有且只有一个主入口；允许在列表/新增/编辑页保留上下文操作，但不能再做成首页/工作台/我的的重复入口。
- 本轮 UI 交互范围统一为列表、新增、编辑。项目整体模块不再输出独立详情页效果；没有特殊业务要求的模块，第三阶段也不新增详情页视觉。
- 通用管理模块统一使用同一套列表、新增、编辑和删除交互：列表页包含标题、标题栏搜索图标、可选筛选、列表项、右上新增；新增/编辑页包含标题左侧返回、字段表单、底部“取消 + 保存/保存修改”；删除统一从列表项左滑触发，并使用小弹窗二次确认。
- 所有页面主标题下方不展示描述性副标题；说明性文案只写入 Handoff/全局规范，不进入 1:1 界面稿。

### 3.3 阶段交付规则

- 第一阶段只冻结业务范围、页面职责、入口边界和验收标准。
- 第二阶段只输出 UI 规范、线框和高保真，不改业务边界。
- 第三阶段只按确认稿编码，不擅自增删入口和文案。
- 任一阶段未确认，不进入下一阶段。

## 4. 功能详细需求

<a id="4-feature-details"></a>

### 4.1 用户认证

#### 4.1.1 邮箱密码登录
- **需求**: 用户使用邮箱和密码登录
- **输入**: 邮箱、密码
- **输出**: 登录成功进入首页，失败显示错误提示
- **异常**: 账号不存在/密码错误时给出明确提示
- **会话**: 登录态持久化，下次打开应用自动登录

#### 4.1.2 邮箱密码注册
- **需求**: 新用户通过邮箱和密码注册
- **输入**: 邮箱、密码（确认密码可选）
- **输出**: 注册成功自动登录，失败显示错误提示
- **异常**: 邮箱已注册时提示

#### 4.1.3 密码重置
- **需求**: 忘记密码时可申请重置
- **流程**: 输入邮箱 → 发送重置邮件 → 用户点击邮件链接重置密码

#### 4.1.4 OAuth 第三方登录
- **需求**: 支持第三方平台登录（如微信、Google）
- **流程**: 点击登录 → 跳转授权页 → 回调 → 自动登录
- **当前状态**: 基础框架已实现，具体 provider 待配置

#### 4.1.5 登出
- **需求**: 用户可安全退出登录
- **行为**: 清除本地 token，跳转到登录页
- **确认方式**: 使用应用内小弹窗二次确认，不使用原生 confirm/Alert 作为视觉方案

#### 4.1.6 路由守卫
- **未登录**: 自动跳转到登录页
- **已登录**: 访问认证页面自动跳转到首页

### 4.2 首页

#### 4.2.1 问候语
- 根据当前时间段显示不同问候语
  - 00:00-12:00: "早上好"
  - 12:00-18:00: "下午好"
  - 18:00-24:00: "晚上好"
- 副标题固定: "今天也要加油哦"

#### 4.2.2 通知入口
- 右上角铃铛图标，有新通知时显示红点
- 点击进入通知中心

#### 4.2.3 数据统计卡片
- **物品**: 显示物品总数，点击跳转到 `/item/list`
- **待办**: 显示待完成数量，点击跳转到 `/todo/list`
- **完成**: 显示已完成数量，不可点击
- 三张卡片只展示关键数字，不展示完整列表、趋势图或二级筛选，避免与列表页/统计页重复

#### 4.2.4 快捷操作

#### 4.8.4 实时推送通知（Socket）
- **后端调度器**: `ReminderScheduler` 服务每分钟检查一次 `life_todos` 表中 `reminder_date` 已到且未完成的待办
- **防重复机制**: 使用内存 Set 记录已发送的提醒（`todo.id + reminder_date`），避免重复推送
- **Socket 事件**: 后端通过 `reminders:fired` 事件向对应用户房间推送提醒数据
- **前端监听**: `notificationStore` 监听 `reminders:fired` 事件，自动将提醒添加到通知中心列表并触发铃铛抖动
- **连接管理**: 用户登录后 `socketService.connect(userId)` 加入 `user:${userId}` 房间，登出时断开

#### 4.8.5 数据持久化

### 4.11 消息模块（v1.4.1 好友验证与共享收拢）

#### 4.11.1 消息列表
- **列表构成**:
  - 好友消息：仅展示已通过好友验证的用户。
  - 系统通知：好友申请、好友申请结果、到期提醒、同步结果等。
- **消息列表项**: 真实头像或头像占位 + 用户名/系统通知名称 + 最后一条消息摘要 + 时间；未读状态使用小红点，不显示数字。
- **消息类型图标**: 使用线性 SVG 图标，不使用含义不明的装饰图标。
- **下拉刷新**
- **添加入口**: 右上角加号打开“添加好友”页，搜索用户并发送验证申请。

#### 4.11.2 添加好友与验证申请
- **搜索用户**: 输入邮箱或用户名，模糊匹配已有用户。
- **发送申请**: 选择用户后填写验证消息并发送。
- **建聊规则**: 对方同意申请前不得建立聊天会话；申请状态通过系统通知反馈。
- **空状态**: 无搜索结果时使用统一空状态组件。

#### 4.11.3 好友操作
- **置顶 / 取消置顶**: 控制好友在消息列表中的排序。
- **共享设置**: 对单个好友设置物品/待办的查看或编辑权限；共享管理不再作为工作台独立入口。
- **删除好友**: 从好友列表左滑触发删除；删除好友关系前必须使用小弹窗二次确认；删除后聊天入口和对该好友的共享设置不可继续操作。
- **扩展预留**: 好友操作页保留后续扩展位置，例如备注名、消息免打扰、黑名单等，但本轮设计稿只展示已确认能力。

#### 4.11.4 聊天页
- **物品卡片消息**: 缩略图 + 名称 + 位置 + 有效期 + "打开"按钮（有权限时进入编辑页，无权限时打开只读摘要 Sheet）
- **待办卡片消息**: 标题 + 优先级标签 + "标记完成"按钮
- **文字消息**: 气泡样式，区分发送者/接收者
- **系统通知消息**: 灰色背景，无发送者气泡
- **聊天输入组件**: 类 IM 输入栏，左侧为消息输入区，右侧为发送按钮，可扩展快速分享物品/待办入口
- **消息排序**: 按 created_at 升序（最新消息在下）

#### 4.11.5 分享联动
- 分享集中在消息好友关系内：
  ```
  物品/待办编辑页 → 选择共享 → 选择已通过好友 →
    1. 创建或更新 share 记录
    2. 写入目标好友权限
    3. 在该好友对话中创建 item/todo 类型消息（卡片数据）
    4. 将 conversation_id 写入 share 记录
  ```
- 对方收到通知后点击 → 打开 app → 直接进入消息 Tab → 进入该对话

#### 4.11.6 搜索用户 API
- **接口**: GET /api/users/search?q=xxx
- **参数**: q — 搜索关键词（邮箱前缀或用户名）
- **返回**: 用户列表 `{ id, email, display_name, avatar_url }`
- **用途**: 添加好友页搜索目标用户

#### 4.11.7 好友申请 API
- **发送申请**: POST /api/friends/requests
- **处理申请**: PUT /api/friends/requests/:id
- **参数**: `{ target_user_id: string, message?: string }`
- **返回**: 申请记录、申请状态和系统通知数据
- **用途**: 好友验证通过后才允许创建聊天会话

#### 4.11.9 数据库表

```sql
CREATE TABLE IF NOT EXISTS life_friend_requests (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  requester_id UUID NOT NULL REFERENCES auth.users(id),
  target_user_id UUID NOT NULL REFERENCES auth.users(id),
  message TEXT,
  status VARCHAR(20) NOT NULL DEFAULT 'pending',
  created_at TIMESTAMPTZ DEFAULT NOW(),
  updated_at TIMESTAMPTZ DEFAULT NOW()
);

CREATE TABLE IF NOT EXISTS life_friend_relations (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID NOT NULL REFERENCES auth.users(id),
  friend_user_id UUID NOT NULL REFERENCES auth.users(id),
  pinned BOOLEAN NOT NULL DEFAULT FALSE,
  created_at TIMESTAMPTZ DEFAULT NOW(),
  updated_at TIMESTAMPTZ DEFAULT NOW(),
  UNIQUE(user_id, friend_user_id)
);

CREATE TABLE IF NOT EXISTS life_conversations (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  participant_ids UUID[] NOT NULL,
  last_message_type VARCHAR(20),
  last_message_content TEXT,
  last_message_at TIMESTAMPTZ,
  created_at TIMESTAMPTZ DEFAULT NOW(),
  updated_at TIMESTAMPTZ DEFAULT NOW(),
  CHECK (array_length(participant_ids, 1) = 2)
);

CREATE TABLE IF NOT EXISTS life_messages (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  conversation_id UUID REFERENCES life_conversations(id) ON DELETE CASCADE,
  sender_id UUID NOT NULL REFERENCES auth.users(id),
  type VARCHAR(20) NOT NULL,
  resource_type VARCHAR(20),
  resource_id UUID,
  content TEXT,
  card_data JSONB,
  created_at TIMESTAMPTZ DEFAULT NOW()
);

CREATE INDEX IF NOT EXISTS idx_messages_conversation ON life_messages(conversation_id, created_at DESC);
CREATE INDEX IF NOT EXISTS idx_messages_sender ON life_messages(sender_id);

-- shares 表保留为权限数据表，但入口收拢到消息好友操作和资源编辑上下文。
ALTER TABLE life_shares ADD COLUMN IF NOT EXISTS conversation_id UUID REFERENCES life_conversations(id);
```

## 6. 交互规范

<a id="6-interaction-specs"></a>

### 6.1 页面导航
- Tab 导航: 底部 4 个 Tab（首页、工作台、消息、我的）
- 工作台: 功能卡片矩阵（3 列网格），点击卡片跳转二级功能页面
- 二级页面: 物品列表页(/item/list)、待办列表页(/todo/list)等，含完整搜索/筛选/排序
- Stack 导航: 列表、新增、编辑页作为子页面推入导航栈；历史详情路由仅做兼容入口
- 返回按钮: 新增/编辑/操作页标题左侧返回图标
- 手势返回: 支持侧滑返回（原生平台）

### 6.2 操作反馈
- **成功**: Toast 提示，绿色，1.5 秒后自动消失
- **错误**: 页面内错误提示、Toast 或小弹窗，根据阻断程度选择
- **加载**: Loading 遮罩或骨架屏
- **确认**: 应用内小弹窗（删除、退出、清理数据、删除好友等敏感操作），不跳转到独立确认页

### 6.3 手势交互
- 左滑删除: 所有可删除的列表项，包括物品、待办、分类、位置、模板、借用、好友等
- 下拉刷新: 所有列表页
- 侧滑返回: 子页面
- 点击复选框: 快速切换待办完成状态

### 6.3.1 搜索组件
- 搜索默认展示为标题栏右侧线性图标，与通知、添加等图标动作同级。
- 页面常规状态不展示常驻搜索输入框，避免占用列表和操作空间。
- 点击搜索图标后进入搜索态、搜索页或搜索 Sheet；只有主动搜索态才展示输入框。

### 6.4 空状态
- 所有列表页必须有空状态引导
- 包含图标、标题、描述、操作按钮
- 按钮引导用户创建第一条数据

### 6.5 表单交互
- 必填字段标记红色星号
- 实时表单验证，聚焦时清除错误
- 提交时全局验证
- 新增/编辑页标题左侧统一为返回图标，底部固定“取消 + 保存/保存修改”
- 删除不放在新增/编辑页，不写“二次确认”等说明性文案到页面；统一从列表项左滑露出删除按钮，点击后打开小弹窗确认
- 键盘自动处理（KeyboardAvoidingView）

### 6.6 加载状态
- 首次加载: 骨架屏（3 条）
- 刷新: 下拉刷新指示器
- 提交: 按钮 loading 状态

## 8. 技术约束

<a id="8-technical-constraints"></a>

### 8.1 平台适配
- 所有功能必须同时适配 **Web (PWA)** 和 **Android**
- 不能使用仅单端可用的方案
- Web 端不支持: `expo-image-manipulator`（压缩）、`expo-notifications`（推送）
- Web 端需使用替代方案

### 8.2 数据库约束
- 所有表使用 `life_` 前缀
- 开发和生产连接同一张表
- 所有表必须启用 RLS
- 分类/位置的 `user_id` 为 NULL 表示系统预设

### 8.3 认证约束
- 认证操作使用 anon key
- service_role key 仅用于管理员操作
- Token 存储在本地（AsyncStorage / localStorage）

### 8.4 API 约束
- 所有接口（除登录/注册）需 Bearer Token 认证
- user_id 由后端从 token 自动注入
- 错误统一格式: `{ message: string }`

### 8.5 文件上传约束
- 单文件 / 批量上传（最多 10 个）
- 存储桶: `items-images` (public)
- 路径格式: `{user_id}/{timestamp}_{filename}`
- Web 端跳过图片压缩

### 8.6 性能约束
- 列表数据考虑分页和缓存
- 搜索使用 300ms 防抖
- 图片上传前压缩（原生平台）

### 8.7 环境变量约束
- 开发环境需清空 `ICUBE_PROXY_HOST`（避免 Supabase 连接超时）
- 敏感信息不得硬编码

### 8.8 表名规范
- 后端表名使用复数: `life_categories`, `life_locations`
- 前端类型名使用单数: `LifeCategory`, `LifeLocation`
