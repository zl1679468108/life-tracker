## 1. 产品定位

<a id="1-product-positioning"></a>

LifeTracker 是一款**个人生活管理工具**，帮助用户追踪物品位置、管理待办事项，通过分类和位置体系实现生活条理化。

**核心价值主张**:
- 找东西不再翻箱倒柜——记录物品存放位置
- 待办不再遗漏——设置优先级和提醒
- 数据可视化——通过统计了解自己的生活模式

**竞品定位**: 介于备忘录类应用（Notion、备忘录）和物品管理类应用之间，轻量级、开箱即用、无需配置。

| 场景 | 描述 |
|------|------|

## 4. 功能详细需求

<a id="4-feature-details"></a>

### 4.1 用户认证

#### 4.1.1 邮箱密码登录
- **需求**: 用户使用邮箱和密码登录
- **输入**: 邮箱、密码
- **输出**: 登录成功进入首页，失败显示错误提示
- **异常**: 账号不存在/密码错误时给出明确提示
- **会话**: 登录态持久化，下次打开应用自动登录

#### 4.1.2 邮箱密码注册
- **需求**: 新用户通过邮箱和密码注册
- **输入**: 邮箱、密码（确认密码可选）
- **输出**: 注册成功自动登录，失败显示错误提示
- **异常**: 邮箱已注册时提示

#### 4.1.3 密码重置
- **需求**: 忘记密码时可申请重置
- **流程**: 输入邮箱 → 发送重置邮件 → 用户点击邮件链接重置密码

#### 4.1.4 OAuth 第三方登录
- **需求**: 支持第三方平台登录（如微信、Google）
- **流程**: 点击登录 → 跳转授权页 → 回调 → 自动登录
- **当前状态**: 基础框架已实现，具体 provider 待配置

#### 4.1.5 登出
- **需求**: 用户可安全退出登录
- **行为**: 清除本地 token，跳转到登录页
- **平台差异**: Web 端使用原生 confirm 弹窗，移动端使用原生 Alert

#### 4.1.6 路由守卫
- **未登录**: 自动跳转到登录页
- **已登录**: 访问认证页面自动跳转到首页

### 4.3 物品管理

#### 4.3.1 物品列表
- 显示所有物品的列表
- 每张卡片显示: 分类图标、名称、描述、位置标签、分类标签
- 左滑显示删除按钮
- 下拉刷新

#### 4.3.2 搜索
- 支持搜索物品名称和描述
- 300ms 防抖延迟
- 聚焦时搜索框高亮（主色边框）
- 有内容时显示清除按钮
- 回车触发搜索

#### 4.3.3 分类筛选
- Chips 横向滚动
- 预设分类: 全部、电子产品、书籍、日用品、衣物、其他
- 选中状态高亮
- 支持用户自定义分类的筛选

#### 4.3.4 排序
- 底部弹出选择器
- 排序选项:
  - 添加时间（默认，最新在前）
  - 名称（A-Z）
  - 分类
- 选中项高亮显示

#### 4.3.5 批量模式
- 点击多选图标进入批量模式
- 显示批量操作栏（全选、已选数量、删除按钮）
- 每个物品卡片显示复选框
- 选中卡片有主色边框高亮
- 点击全选/取消全选
- 批量删除需要二次确认

#### 4.3.6 创建/编辑物品
- **必填字段**:
  - 物品名称（左图标 tag）
  - 分类（网格选择，带图标）
  - 存放位置（网格选择，带图标）
- **选填字段**:
  - 图片（最多 5 张，支持拍照/相册）
  - 描述（多行文本）
- **表单验证**:
  - 名称为空时提示"请输入物品名称"
  - 未选择分类时提示"请选择分类"
  - 未选择位置时提示"请选择存放位置"
  - 实时清除错误提示
- **保存按钮**: 导航栏右侧"保存"按钮
- **创建和编辑复用**: 同一页面，通过 URL 参数区分

#### 4.3.7 物品详情
- 顶部: 分类渐变图标 + 名称 + 描述
- 信息卡片: 分类和位置（图标 + 名称）
- 图片: 横向滚动，点击可全屏预览
- 时间: 添加时间和更新时间
- 编辑按钮: 导航栏右侧
- 删除按钮: 底部红色按钮，需要确认

#### 4.3.8 图片处理
- 选择: 拍照或从相册选择
- 压缩: 原生平台压缩至 1024px 宽，质量 0.8
- Web 端跳过压缩（不支持 ImageManipulator）
- 上传: 调用后端 API 上传到 Supabase Storage
- 预览: 全屏查看，支持滑动切换
- 删除: 点击图片预览中的删除按钮移除

### 4.8 通知中心

#### 4.8.1 通知列表
- **待办提醒**: 显示未完成的待办（前3条），包含截止日期
- **物品统计**: 显示当前物品总数
- **系统通知**: 欢迎消息等
- **推送通知**: 待办提醒时间到达时自动添加到通知列表
- **已读/未读状态**:
  - 未读通知有橙色浅背景 + 橙色边框高亮
  - 未读通知标题加粗，右侧显示红色圆点
  - 点击未读通知自动标记为已读
- **全部标记已读**: 未读 tab 下显示"全部标记为已读"按钮

#### 4.8.2 Tab 筛选
- **全部**: 显示所有通知
- **未读**: 仅显示未读通知
- **已读**: 仅显示已读通知
- Tab 切换使用分段控制器样式

#### 4.8.3 首页铃铛动画
- 有未读通知时，铃铛图标执行抖动动画（左右摇摆 2 次）
- 推送通知到达时，铃铛重新触发抖动动画
- 有未读通知时，铃铛右上角显示红色圆点
- 无未读通知时，铃铛静止且无红点

#### 4.8.4 实时推送通知（Socket）
- **后端调度器**: `ReminderScheduler` 服务每分钟检查一次 `life_todos` 表中 `reminder_date` 已到且未完成的待办
- **防重复机制**: 使用内存 Set 记录已发送的提醒（`todo.id + reminder_date`），避免重复推送
- **Socket 事件**: 后端通过 `reminders:fired` 事件向对应用户房间推送提醒数据
- **前端监听**: `notificationStore` 监听 `reminders:fired` 事件，自动将提醒添加到通知中心列表并触发铃铛抖动
- **连接管理**: 用户登录后 `socketService.connect(userId)` 加入 `user:${userId}` 房间，登出时断开

#### 4.8.5 数据持久化
- 已读状态通过 AsyncStorage 本地持久化
- 应用重启后保持已读状态

#### 4.8.6 空状态
- 无通知时显示铃铛关闭图标 + "暂无通知"
- Tab 筛选无结果时显示对应提示（"暂无未读通知" / "暂无已读通知"）

## 8. 技术约束

<a id="8-technical-constraints"></a>

### 8.1 平台适配
- 所有功能必须同时适配 **Web (PWA)** 和 **Android**
- 不能使用仅单端可用的方案
- Web 端不支持: `expo-image-manipulator`（压缩）、`expo-notifications`（推送）
- Web 端需使用替代方案

### 8.2 数据库约束
- 所有表使用 `life_` 前缀
- 开发和生产连接同一张表
- 所有表必须启用 RLS
- 分类/位置的 `user_id` 为 NULL 表示系统预设

### 8.3 认证约束
- 认证操作使用 anon key
- service_role key 仅用于管理员操作
- Token 存储在本地（AsyncStorage / localStorage）

### 8.4 API 约束
- 所有接口（除登录/注册）需 Bearer Token 认证
- user_id 由后端从 token 自动注入
- 错误统一格式: `{ message: string }`

### 8.5 文件上传约束
- 单文件 / 批量上传（最多 10 个）
- 存储桶: `items-images` (public)
- 路径格式: `{user_id}/{timestamp}_{filename}`
- Web 端跳过图片压缩

### 8.6 性能约束
- 列表数据考虑分页和缓存
- 搜索使用 300ms 防抖
- 图片上传前压缩（原生平台）

### 8.7 环境变量约束
- 开发环境需清空 `ICUBE_PROXY_HOST`（避免 Supabase 连接超时）
- 敏感信息不得硬编码

### 8.8 表名规范
- 后端表名使用复数: `life_categories`, `life_locations`
- 前端类型名使用单数: `LifeCategory`, `LifeLocation`
