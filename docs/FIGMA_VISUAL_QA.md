# LifeTracker Visual QA & Polish Plan

> 状态: Figma 流程暂停；当前审查对象为本地 HTML/CSS/JS/SVG 高保真设计源  
> 日期: 2026-06-28  
> 目标: 用 `ui-ux-pro-max` 与 `frontend-design` 标准复查本地完整交互地图，让后续 1:1 前端还原更稳定、更高保真。  
> 当前设计源: `docs/design/lifetracker-v1.4-complete-interaction-map.html`

## 1. 设计审查原则

- 产品定位: 生活效率工具 + 轻量操作台，优先可扫描、可操作、低认知负担。
- 导航不变: 底部固定 `首页 / 工作台 / 消息 / 我的`，不新增一级 Tab。
- 入口不重复: 工作台平铺业务模块；我的平铺账号、偏好、数据与支持。
- 页面形态不扩张: 无特殊要求时只保留 `列表 / 新增 / 编辑`，不新增详情页效果。
- 深浅色同等重要: 深色不是唯一稿，浅色要有完整背景、表面、文字、边框、阴影和语义色映射。
- 图标规范: 线性图标优先；无法准确表达时使用模块名称单字图标；不得用模糊装饰色块冒充图标。
- 触控与可访问性: 所有可点击区域不小于 `44 x 44`，正文对比度不低于 4.5:1。
- 代码还原友好: 所有关键屏应有明确尺寸、间距、字号、行高、圆角、边框、状态和组件命名。

## 2. 全局高保真优化项

| 项目 | 当前风险 | 优化要求 |
|---|---|---|
| 组件命名 | Figma 节点可读，但部分模板还偏说明稿 | 补充 `Component Spec / Row / Button / Field / Sheet / Dialog` 命名区 |
| 设计 token | 已有色彩，但缺少一眼可抄的实现表 | 在 Figma 增加 `Implementation Tokens` 表，记录 RN 常量名 |
| 图标 | 目前部分用单字图标 | 后续补充线性 SVG 图标样式示例，单字只作 fallback |
| 状态 | 覆盖规则已写，视觉状态不足 | 增加 loading / empty / error / selected / disabled / danger confirmation 小样 |
| 表单 | 有新增/编辑模板，但校验态不足 | 增加 required、helper、error、disabled、saving 状态 |
| 列表 | 有列表模板，但密度和批量状态需更明确 | 增加普通行、选中行、滑动删除、空列表、搜索无结果 |
| 浅色模式 | 已补浅色模板，但数量少于深色 | 首页、工作台、列表、我的需有可对照浅色关键屏 |
| 开发标注 | 目前散落在文档和 Figma | 在 Figma 补 `Dev Handoff Overlay`，集中写尺寸和间距 |

## 3. 逐屏优化建议

### 3.1 首页

- 保留总览职责，不展示所有功能入口。
- 搜索默认只作为标题栏图标入口；常规页面不常驻搜索输入框。
- 统计卡避免过度装饰，三项指标使用统一数字样式 `28/34/800`。
- 快捷动作只保留添加物品、添加待办；通知入口保持右上角图标按钮。
- 最近待办最多 4 条，补充空状态和 loading 骨架。
- 浅色模式需要与深色同构，不能只换背景色。

### 3.2 工作台

- 工作台是全量业务入口，不设“更多工具”。
- 工作台不展示 `我的` 入口；`我的` 只保留在底部 Tab 和账号/系统设置页。
- 分组顺序固定: `核心功能 / 管理工具 / 生活记录 / 数据与提醒`。
- 每个入口行或卡片必须包含: 图标、标题、说明、chevron、点击范围。
- 避免卡片宫格过密；移动端优先列表或两列卡片，不超过用户一屏扫描负担。
- 入口高度建议 `66-76`，分组标题 `16/24/700`，说明 `11/16/400`。
- 补充搜索功能的筛选态: 搜索“数据”时命中统计、通知、数据管理、资产、小组件。

### 3.3 物品列表 / 新增 / 编辑

- 列表行需要明确图片/图标占位、物品名称、分类、位置、价值或保质期、chevron。
- AI 识别作为新增/编辑上下文能力，不做独立入口。
- 新增页顶部应有图片/条码/AI 识别区域，下面是表单字段。
- 编辑页应与新增页共用表单模板，删除不进入编辑页。
- 状态补齐: 图片上传中、识别建议、必填错误、保存中、列表左滑删除确认。

### 3.4 待办列表 / 新增 / 编辑

- 列表行包含 checkbox、标题、截止日期、优先级、分类/关联物品。
- 分段控制器固定: `全部 / 待完成 / 已完成`。
- 拖拽排序与完成态要有可见差异，不能只依赖颜色。
- 新增/编辑字段: 标题、截止时间、优先级、提醒、关联物品、备注。
- 状态补齐: 逾期、今日、已完成、空列表、筛选无结果。

### 3.5 消息

- 消息列表要明确未读小红点、时间、资源卡片摘要。
- 加号进入添加好友页，包含搜索用户和验证消息；对方同意前不创建聊天。
- 好友操作页包含置顶/取消置顶、共享设置；删除好友从好友列表左滑触发。
- 对话页如果保留，需要作为消息特殊页处理，不扩散成其他模块详情页。
- 状态补齐: 无对话、搜索无用户、发送中、发送失败、分享卡片。

### 3.6 我的

- 我的页外层平铺，不再把同类入口藏到二级大卡内。
- 分组固定: `账号 / 偏好 / 数据与支持`。
- 退出登录必须视觉上与普通入口分离，并使用危险色。
- 数据同步需要 loading / success / error 三态。
- 版本信息为 readonly row，不能表现成可编辑入口。

### 3.7 管理模块模板

- 覆盖分类、位置、模板三类管理模块。
- 默认页面形态: 列表 / 新增 / 编辑。
- 分类和位置支持层级时，列表行要表现父子关系缩进。
- 模板管理需补充“套用模板”入口和“保存为模板”上下文说明。
- 状态补齐: 空列表、新增成功、编辑保存、列表左滑删除确认。

### 3.8 生活记录模板

- 覆盖借用、日历；共享集中到消息好友操作和资源编辑上下文。
- 借用管理默认列表 / 新增 / 编辑，包含借用人、物品、归还日期、状态。
- 日历视图可保留月视图 + 当天提醒列表，不增加复杂详情页。
- 状态补齐: 逾期、即将归还、无事件、权限只读。

### 3.9 数据与提醒模板

- 覆盖数据统计、通知中心、数据管理、资产总览、桌面小组件。
- 图表不追求复杂装饰，优先清晰趋势和直接数值。
- 通知中心分段: `全部 / 未读 / 已读`。
- 数据管理必须将导出、导入、清理分为普通动作和危险动作。
- 桌面小组件作为设置说明 + preview，不做独立复杂编辑器。

## 4. 本地设计源审查清单

当前不再写入 Figma，执行以下本地设计源审查:

1. `行 = 功能大类，列 = 子功能/交互页` 排列完整画板。
2. 第 1 行: 首页。
3. 第 2 行: 工作台、搜索结果列表。搜索结果必须展示管理工具、生活记录、数据与提醒的全部命中入口。
4. 第 3 行: 物品列表、物品新增、物品编辑。
5. 第 4 行: 待办列表、待办新增、待办编辑。
6. 第 5 行: 消息列表、添加好友、好友操作、对话页。
7. 后续行依次补齐我的、分类、位置、模板、借用、日历、数据统计、通知中心、数据管理、资产总览、桌面小组件。
8. Handoff 区包含 `Implementation Tokens`、图标库、交互组件库、通用 CRUD、状态规则。
9. 浅色模式至少覆盖首页、工作台/搜索、我的、数据类关键屏。

## 5. 截图审查结论

基于用户导出的 Figma 截图，当前设计源需要以下修正：

- `PRD Full Coverage Supplement` 更像覆盖说明板，不足以作为完整 1:1 还原源。
- 消息、数据统计、通知中心、数据管理、资产、小组件等模块缺少独立功能界面效果。
- 工作台错误展示了 `我的` 入口，应移除。
- 部分圆形图标语义不明确，尤其顶部圆形按钮中的闪电/折线图标；应替换为明确的搜索、通知铃铛、加号、聊天气泡等线性图标。
- 画板排列应从“概览/模板混排”改为“行代表功能大类，列代表列表/新增/编辑/状态”的交互地图。
- 低频模块不能只用模板代替，应至少有列表、新增、编辑或明确的状态页样例。

## 6. Handoff 门禁

- 当前仍处于第二阶段设计补齐。
- 完整交互地图经用户确认前，不进入第三阶段编码。
- 如历史 Figma 与本地设计源不一致，以用户最终确认的本地 HTML/CSS/JS/SVG 设计源为准。
